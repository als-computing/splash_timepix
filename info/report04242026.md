## Problem (1): Trailing-data loss on disconnect

### Symptom

For an acquisition with N TDC pulses (= 2 × N TDC packets sent by the
simulator’s counting CLI), `stop.total_cycles` consistently undercounts by
≈ `flush_interval` × `tdc_frequency_hz`, *not* by the off-by-one I initially
predicted.

Reproducer: spawn `splash_timepix.app` + `splash_timepix.simulator_cli
--auto-start`, parse the simulator’s `"X TDC events"` stdout line, and read
`stop.total_cycles` from the ZMQ `stop` message.

### Root cause

`splash_timepix.app.main` snapshots `cycle_count` into the `TimePixStop`
message **before** `SocketDataServer.message_queue` (the *raw‑byte* queue
between the TCP reader and the parsing/binning processor) has been drained.

Sequence of events at end-of-acquisition (before the fix):

1. Producer (TimePix detector / simulator) sends final batch of TDC bytes,
   then `close()`s the TCP connection.
2. `SocketDataServer._handle_client` reads up to TCP FIN, flushes residual
   bytes onto `message_queue`, and exits.
3. `_socket_listener` sets `client_disconnected_event`.
4. The `app.py` main loop wakes from its 1-second sleep, sees the event, and
   calls `do_final_flush()` followed by `_wait_for_xyt_drain()`.
   `_wait_for_xyt_drain()` waits for the **output** queue (`xyt_queue`,
   array+metadata to ZMQ) to drain — but **not** for the **input** queue
   (`message_queue`).
5. `cycle_count` is read into `TimePixStop.total_cycles`.
6. `data_processor` continues to drain `message_queue` after the stop is
   queued. Those late `data_callback` invocations *would* have called
   `handle_tdc` and incremented `cycle_count`, but the snapshot has already
   been taken. The `xyt_queue` has been drained, the `zmq_worker` has been
   told to shut down, so any flush events these late TDCs trigger never make
   it onto the wire either.

Net effect: roughly `flush_interval` × `tdc_frequency_hz` cycles silently
disappear from both `stop.total_cycles` *and* the published events.

The relevant pre-fix code path is `app.py` lines 661–772 (`exit_on_disconnect`
branch) and lines 747–772 (natural-disconnect branch). Both go straight into
`do_final_flush()` without ever waiting on the producer→processor queue.

### Fix

Two parts:

1. New helper on `SocketDataServer` that uses the queue’s existing task
   counter (`unfinished_tasks` / `all_tasks_done`, populated by the
   `task_done()` calls already in `_data_processor`) to block until every
   batch currently in `message_queue` has returned from `data_callback`:

```189:213:src/splash_timepix/socket_server.py
    def wait_for_idle(self, timeout: float = 5.0) -> bool:
        """Block until every batch currently in ``message_queue`` has been processed.

        The data processor calls ``message_queue.task_done()`` after each
        batch returns from ``data_callback``, so when ``unfinished_tasks``
        reaches zero we know the callback has been invoked for every batch
        the socket reader put in.

        Intended to be called *after* ``_handle_client`` has exited (i.e.
        the client has disconnected and no further batches will be put);
        otherwise this can wait indefinitely for new producer activity.

        Returns:
            True if the queue drained within ``timeout``, False on timeout.
        """
        deadline = time.monotonic() + timeout
        with self.message_queue.all_tasks_done:
            while self.message_queue.unfinished_tasks > 0:
                remaining = deadline - time.monotonic()
                if remaining <= 0:
                    return False
                self.message_queue.all_tasks_done.wait(timeout=remaining)
        return True
```

2. `app.py` calls `server.wait_for_idle(timeout=5.0)` *before* `do_final_flush()`
   on **all three** stop sites:

   - the `exit_on_disconnect` branch (around lines 661–700),
   - the natural-disconnect branch (around lines 747–785),
   - the `finally` block (around lines 921–940).

Each call site logs a warning (and proceeds) if the queue fails to drain
within 5 s, so a stuck producer doesn’t hang shutdown forever.

### Verification — gap before vs after the fix

Methodology: run `splash_timepix.app` + `splash_timepix.simulator_cli
--auto-start` with the configuration given by `tdc_hz` × `dur`, parse the
simulator’s `"<N> TDC events"` print to count TDC packets emitted, divide by 2
to get pulse count, and compare against (a) `stop.total_cycles` and (b)
`Σ cycles_in_flush` over event messages received before the stop.

Predicted (off-by-one only): `stop.total_cycles == sim_pulses − 1`.

**Before the fix**

| 10  |  5 |  49 |  42 |  47 | 14.3 % |

shrinks from ~5 % to <1 % and continues to shrink with duration, which is the
behaviour we expect once the trailing-data race is closed.

### Does this affect real DAQ?

Yes.

- The fix is in `splash_timepix.app`, which is the *same* server `splash_timepix.ui.main`
  spawns in production. Production runs against Serval + luna-iterator (per
  `solution.md` and the bursty-wire test) which batches TDCs upstream into
  multi-second boluses, so a non-trivial number of TDCs are typically still
  in `message_queue` at the moment the upstream connection closes. Without
  this fix that bolus is silently lost.
- For long acquisitions (minutes / hours) the fractional loss is small, but
  the *absolute* loss is `≈ flush_interval × tdc_frequency_hz` cycles **on
  every acquisition end** regardless of run length — a non-zero data integrity
  issue, not just an accounting issue.
- Anyone using `stop.total_cycles` (or the implicit `Σ cycles_in_flush` UI
  running average) for normalisation was previously normalising to a
  systematically low denominator. The numerator (binned counts) was *also*
  systematically low because the same TDCs never made it through
  `data_callback`, so the running average was approximately self-consistent —
  but every acquisition was missing its trailing window of physics.

Recommended action for ongoing real-DAQ work: pull this fix and treat
acquisitions ended pre-fix as having a missing trailing `flush_interval`.

## Problem (2) — caveat: connect-time orphan flush

This is a **pre-existing** bug found while validating the fix above. It is
**not** addressed in this session.

### Symptom

In some runs the server log shows a `Flushed: #1, cycles=N` event published
*before* the per-connection reset block runs:

```
16:22:36,365  INFO  Flushed: #1, cycles=5
16:22:36,402  INFO  New client connected - resetting acquisition state.  New scan: 2736d984-…
16:22:36,402  INFO  Queued start message on connect for scan: 2736d984-…
16:22:36,973  INFO  Flushed: #1, cycles=5            ← real flush #1, post-reset
```

The orphan event carries real binned data (5 cycles worth in this example) but
its `scan_name` is whatever was set before the reset — the previous
acquisition’s UUID, or `None` for the very first connection. After the reset
runs, `cycle_count` is wiped to 0, so the cycles in the orphan flush are
**not** counted in the eventual `stop.total_cycles`.

This is what makes `Σ cycles_in_flush > stop.total_cycles` in the post-fix
table above. The events are fine — *all* the data is on the wire — but the
stop snapshot lies.

### Root cause

`SocketDataServer._data_processor` is a daemon thread that starts when the
server starts; it runs independently of the `app.py` main loop. As soon as a
client `connect()`s, bytes start flowing into `message_queue`,
`_data_processor` parses them, calls `data_callback`, which calls
`handle_tdc`, which can call `emit_flush_if_due`, which emits a flush event
the moment the wall-clock gate (`flush_interval` since the first TDC) opens.

Meanwhile, the main loop is in `time.sleep(1)`. When it wakes, it sees
`server.client_connected and not was_client_connected` and runs the
per-acquisition reset:

```703:728:src/splash_timepix/app.py
                # ── Authoritative per-acquisition reset ──────────────────────
                # scan_name is generated here and nowhere else (single source of truth).
                # All counters declared above are also reset here for each new client.
                stop_message_sent_on_disconnect = False
                start_message_sent = False
                acquisition_start_time = None
                scan_name = generate_scan_name()
                cycle_count = 0
                flush_count = 0
                event_count = 0
                pixels_before_trigger = 0
                pixels_outside_window = 0
                t_zero = None
                first_pixel_time = None
                # Wall-clock gate: clear so the first TDC of the new
                # acquisition re-arms the clock (instead of carrying
                # over a stale t_last from the previous client).
                last_flush_time = None
                cycles_since_last_flush = 0
                # Clear arrays for fresh start
                with xyt_lock:
                    xyt_array.fill(0)
                local_accumulator.fill(0)
```

If the producer’s first `flush_interval` of TDCs all arrive and process
*before* the main loop wakes, an emit fires under the *previous* acquisition’s
state, *then* the reset runs and zeroes everything. The data is on the wire
under the wrong tag.

Whether the orphan event appears is purely a function of timing: how quickly
the producer ramps up vs. when the main-loop sleep happens to wake. The
bursty-wire production profile (Serval + luna-iterator boluses) is more
likely to hit it than the steady simulator, because the first bolus is
typically several `flush_interval`s of pre-batched data.

### Does this affect real DAQ?

Yes.

- Same race window applies to the production server, with the additional
  twist that `solution.md`’s bursty-wire profile front-loads the first ~2 s of
  an acquisition into a single TCP `sendall`, which is exactly the shape that
  fills the orphan window before the main loop notices.
- Subscribers (the operator UI, anything else listening) that filter events
  by `scan_name` against the most recent `start` will silently drop the
  orphan event. The orphan event also predates the `start` message in the
  ZMQ stream, so any subscriber that gates on having seen a `start` first
  will drop it on those grounds alone.
- `stop.total_cycles` will undercount the leading window even with problem
  (1) fixed.

### Recommended fix (not done)

Two reasonable options, in increasing order of intrusiveness:

1. **Move the per-connection reset out of the main loop and into a
   connection-accepted callback.** `_socket_listener` already knows the
   exact moment a `client_socket` is accepted — that is the right place to
   reset acquisition state, *before* `_handle_client` runs and *before*
   `_data_processor` is allowed to call `data_callback`. The current
   `was_client_connected` polling in the main loop becomes purely
   informational. This fully closes the orphan window.

2. **Gate `data_callback` on a `start_of_acquisition_armed` flag.** Initialize
   the flag to `False`. Set it to `True` from the connection-accepted
   callback (after the reset). Have `data_callback` early-return while the
   flag is `False`. This drops the leading bytes rather than misattributing
   them, which is at least consistent — but it is **data loss**, so option 1
   is preferable.

Either fix should be paired with an integration test that asserts
`Σ cycles_in_flush == stop.total_cycles` for a *single* acquisition end-to-end
(the existing `tests/test_flush_burstiness.py::test_batched_wire_regression`
already asserts this invariant — see lines 685–695 — and currently passes
only because its 15 s duration averages out the orphan and trailing windows
under the failure tolerances; tightening the invariant or shortening the
test duration will surface the regression).
